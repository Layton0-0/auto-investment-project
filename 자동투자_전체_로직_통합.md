# 자동투자 전체 로직 통합 문서

**목적**: 본 프로젝트(하위 포함)의 자동투자 관련 **전체 로직·데이터 흐름·스케줄·구성요소**를 하나의 문서로 정리한다.  
**대상**: investment-backend, investment-data-collector, investment-frontend, investment-infra 및 관련 설정·배치·API.

**상세 전략·수식**: [investment-backend/docs/02-architecture/00-strategy-registry.md](investment-backend/docs/02-architecture/00-strategy-registry.md), [12-auto-investment-strategy.md](investment-backend/docs/02-architecture/12-auto-investment-strategy.md)

---

## 1. 전체 구조 요약

```
[데이터 수집] → [팩터·시그널 계산] → [리스크 게이트·자본 배분] → [진입 실행] → [청산 평가·매도] → [체결·PnL·거버넌스]
     ↑                    ↑                      ↑                    ↑                    ↑
  Python/Backend      Backend 08:00           Backend 09:10         Backend            Backend 장중/매분
  DART/SEC/KRX/US     유니버스·시그널         로보→파이프라인       KR/US×단중장       ExitRule·Fill·DailyPnL
```

- **자동매수(통합)**: 한 번의 실행으로 **로보(ETF) → 파이프라인(개별종목)** 순서 실행. 퀀트 유리 시간대(KR 09:00~10:00, US 23:30~01:00 KST) 내에서만 진입 가능(설정 시).
- **실제 주문 여부**: **DB에서 ADMIN이 선택한 값만** 사용. 시스템 레벨 dry-run 파라미터는 없음. 서버 설정 `pipeline.autoExecute`(TB_SYSTEM_SETTINGS)·계정별 `pipelineAutoExecute`(TB_TRADING_SETTINGS)에 따라 실행/권장만 결정.
- **손익 기준**: 모든 매매 판단·기대수익은 **수수료·세금 포함 순손익(net)** 기준. `FrictionCostService`·왕복 비용률 반영.

## 2. 프로젝트별 역할

| 프로젝트 | 역할 | 자동투자 연관 |
|----------|------|----------------|
| **investment-backend** | Spring Boot API·배치·팩터·파이프라인·주문·청산·로보·리스크·거버넌스 | 핵심. 팩터 계산·자동매수·청산·체결 확인·일일 PnL·전략 거버넌스 |

## 3. 데이터 수집 (파이프라인 입력)

### 3.1 담당·스케줄

| 데이터 | 담당 | 스케줄(기본) | 저장소 |
|--------|------|--------------|--------|
| DART 공시 | Python data-collector | 10분마다 (SCHEDULE_DART_SEC=1) 또는 수동 POST /dart-collect | Spring POST /api/v1/internal/collected-news → TB_NEWS_ITEMS |
| SEC EDGAR 공시 | Python data-collector | 15분마다 또는 수동 POST /sec-collect | 동일 |
| KRX 일별 시세 | Backend KrxCollectionService | 16:00 KST, 수동 /trigger/krx-daily | TB_DAILY_STOCK |
| US 일별 시세 | Backend → Python /us-daily | 17:00 KST, 수동 /trigger/us-daily | TB_DAILY_STOCK |

- Backend 트리거: `POST /api/v1/trigger/krx-daily`, `us-daily`, `dart-collect`, `sec-collect` (실제 DART/SEC는 Backend가 data-collector 호출 또는 별도 스케줄).

### 3.2 데이터 수집 경로 요약

- **DART/SEC**: data-collector `app.py` → `/dart-collect`, `/sec-collect` → Spring `InternalDataCollectionController` → TB_NEWS_ITEMS. 시그널 키워드 매칭 시 `eventType` DART_SIGNAL:%, 8K 등.
- **KRX**: Backend `KrxCollectionService` → KRX Open API (유가증권 일별매매정보) → TB_DAILY_STOCK.
- **US**: Backend가 data-collector `POST /us-daily` 호출 또는 Backend 내부 US 수집 → TB_DAILY_STOCK. 유니버스는 `investment.data.us.symbols`(퀀트 스타일: 지수·섹터 ETF·메가캡).
- **기업 이벤트 제외**: `CorporateActionService.getSymbolsToExclude(basDt, market)` — 액면분할·배당락 등 가격 왜곡 이벤트 종목을 유니버스 필터 단계에서 제외(데이터 소스 연동 전에는 빈 집합).

## 4. 4단계 파이프라인 (Backend)

### 4.1 단계 요약

| 단계 | 내용 | 구현 |
|------|------|------|
| 1단계 유니버스 | 유동성·Sector RS(한국)·Post-Earnings Drift(미국)·기업 이벤트 제외 | UniverseFilterService, CorporateActionService, TB_UNIVERSE |
| 2단계 시그널 | 이격도·변동성 돌파·듀얼 모멘텀·수급·RSI/MACD 등 점수 | FactorCalculationService, TB_SIGNAL_SCORE |
| 3단계 자금 관리 | ATR·Half-Kelly·변동성 역가중·공분산 패널티·수수료 반영 | PositionSizingService, CorrelationPenaltyService, FrictionCostService |
| 4단계 실행·청산 | 권장 종목 주문 실행(단일 또는 Algo)·장중 청산·매도 | PipelineExecutor, PipelineOrderExecutor, ExitRuleService, PipelineExitScheduler |

### 4.2 팩터 계산 (매일 08:00 KST)

- **Batch Job**: `factor-calculation`
- **역할**: 기준일(bas_dt) 기준 유니버스 필터 → 시그널 점수 산출 (TB_SIGNAL_SCORE, 유니버스 필터 결과).
- **트리거**: `POST /api/v1/trigger/factor-calculation`

### 4.3 자동매수(통합) — 09:10 KST

- **Batch Job**: `auto-buy`
- **흐름**: `AutoBuyOrchestrator.run()` (인자 없음). 실제 주문 여부는 DB만 사용.  
  1. **로보(ETF)**: 로보 ON 계좌에 대해 `RoboRebalanceScheduler.runNow()` — 동적 자산배분·리밸런싱(실행 전 백테스트 통과 시 ETF 주문). 주문 실행 여부는 DB `pipeline.autoExecute`에 따름.  
  2. **파이프라인(개별종목)**: 자동투자 ON 계좌에 대해 `PipelineExecutionScheduler.runNow(marketFilter)`. 계좌별 실행 여부는 DB 시스템 설정·계정별 `pipelineAutoExecute`에 따름.

### 4.4 파이프라인 실행 상세 (PipelineExecutionScheduler)

- **대상**: `TradingSetting.autoTradingEnabled == true` 인 계좌만.
- **실행 전 검사 (계좌별)**  
  - 리스크 게이트(`RiskGateService`: 레짐·VIX 등) 통과 여부.  
  - 시장 급락 게이트(`MarketCrashGateService`: 벤치마크 일일 -5% 등) 신규 매수 허용 여부.  
  - 일일 손실 한도(`DailyLossLimitService`) 허용 여부.  
  - 거버넌스 halt(`GovernanceHaltService`: market+strategyType별) 여부.  
  - **퀀트 매매 유리 시간대**: `TradingWindowService.isInKrWindow()` / `isInUsWindow()` — 설정 시 해당 시장은 윈도우 내일 때만 run.
- **자본 배분**: 계좌별 `maxInvestmentAmount`(또는 서버 기본 자본)·`StrategyWeightResolver`로 단기/중기/장기 비율(기본 20/40/40) → shortCapital, midCapital, longCapital. (market, strategyType)별 **CapitalDrawdownConstraintService.getCapitalMultiplier**로 단기 열화 시 배정 자본 절반(0.5) 적용 가능.
- **6회 run**: (KR, SHORT_TERM), (KR, MEDIUM_TERM), (KR, LONG_TERM), (US, SHORT_TERM), (US, MEDIUM_TERM), (US, LONG_TERM).  
  각 run: `PipelineExecutor.run(basDt, market, accountNo, strategyType, allocatedCapital, autoExecute)`. `autoExecute`는 계좌별 DB 값(계정 pipelineAutoExecute 또는 서버 기본 pipeline.autoExecute).
- **실제 주문**: `pipeline.use-algo-execution=true` 시 **PipelineOrderExecutor**(TWAP/VWAP 등 알고리즘 실행), false 시 **OrderService** 단일 주문. DB에서 ADMIN이 선택한 값(시스템 설정 `pipeline.autoExecute`, 계정별 `pipelineAutoExecute`)에 따라 실행. 실전 계좌는 `pipeline.allow-real-execution` 별도 허용 필요.

### 4.5 자동매수(미국장) — 23:35 KST

- **Batch Job**: `auto-buy-us`
- **역할**: 파이프라인 **US 시장만** 실행 (`marketFilter=US`). 미국 유리 시간대(23:30~01:00 KST) 내에서만 진입.

### 4.6 포지션 사이징·수수료 반영 (PositionSizingService, CorrelationPenaltyService, FrictionCostService)

- **진입 게이트**: 2:1 R:R 가정 시 기대 gross 수익률이 **왕복 비용률(수수료·세금·슬리피지)을 초과**할 때만 권장.  
  `FrictionCostService.getRoundTripCostRate(market)` 사용.
- **포트폴리오 상관관계 패널티**: `CorrelationPenaltyService.applyPenalty` — 권장 목록에서 종목 간 상관계수가 임계값(기본 0.7) 이상인 쌍이 있으면 비중을 스케일 다운(기본 0.8). 설정: `investment.factor.correlation-penalty-enabled`(기본 false).
- **기대수익 표시**: 트레이딩 포트폴리오·단기 전략 등에서는 **순손익**(기대수익률 − 왕복 비용률) 기준.

## 5. 청산·체결·사후 처리 (Backend)

### 5.1 청산 평가 (장중 5분마다, 09:00~15:59 평일)

- **Batch Job**: `pipeline-exit`
- **역할**: 보유 포지션(TB_STRATEGY_POSITION)에 대해 `ExitRuleService.getSellSignals(accountNo, currentPriceBySymbol, todayHighBySymbol)` 호출.  
  현재가·당일 고가는 **RealtimeMarketDataService.getCurrentPrices()** 사용. WebSocket이 활성화된 경우 **WebSocket 수신 가격을 우선 반영**(`webSocketLivePrices`·캐시 갱신); 미수신 종목은 REST 기반 5분 캐시. `WebSocketPriceCacheListener`가 실시간 호가/체결 이벤트 수신 시 현재가 캐시 갱신.  
  ATR Trailing Stop, -3% Trailing, -5%/-10% 손절, 전저점 이탈(KR), RSI≥70 익절(KR), Time-Cut(단기 전용) 등 적용.
- **실행**: 매도 시그널 발생 시 `OrderService`로 매도 주문(auto-execute=true일 때).

### 5.2 체결 확인·포지션 등록 (매분)

- **Batch Job**: `fill-confirmation`
- **역할**: 체결된 주문(EXECUTED) → TB_STRATEGY_POSITION 등록 (`PipelineExecutor.registerPositionOnExecution`).  
  `investment.pipeline.register-position-on-execution=true` 이면 체결 확인 후 등록, false면 주문 성공 시 즉시 등록.

### 5.3 미체결 확인·알림 (매분)

- **Batch Job**: `unfilled-order-check`
- **역할**: PENDING N분 경과 시 Discord 긴급 알림 (`EmergencyAlertService`).  
  설정: `alert-discord-webhook-url`, `unfilled-check-minutes`.

### 5.4 일일 PnL 기록 (16:05 KST 평일)

- **Batch Job**: `daily-pnl`
- **역할**: 계좌별 당일 수익률 기록 (`DailyPnlService`).

### 5.5 장중 변동성 돌파 (09:10, 09:40 평일)

- **Batch Job**: `intraday-breakout`
- **역할**: `IntradayBreakoutService`로 당일 시가+Range×k 돌파 종목 후보 산출, 설정 시 진입.  
  현재가는 `RealtimeMarketDataService`(REST+캐시) 사용.

## 6. 리스크·거버넌스 (Backend)

### 6.1 리스크 게이트 (파이프라인 실행 전)

- **RiskGateService**: 레짐·VIX 등으로 신규 매수 허용 여부·비중 배수 결정.  
  `MacroIndicatorProvider`(VIX·거시 지표) 연동.
- **MarketCrashGateService**: 벤치마크(기본 SPY) 전일 일일 낙폭 ≥ N%(기본 5%) 시 당일 신규 매수 중단.
- **DailyLossLimitService**: 당일 시초 평가액 대비 손실이 한도 초과 시 당일 신규 매수 중단.
- **CapitalDrawdownConstraintService**: (market, strategyType)별 단기(5영업일) Hit Ratio·Profit Factor가 장기(60일) 대비 2σ 이상 열화 시 배정 자본을 절반(0.5)으로 제한. 설정: `investment.risk.capital-drawdown-constraint-enabled`(기본 false). 데이터 연동 전에는 1.0 배율.

### 6.2 전략 거버넌스 (매월 1일 02:00 KST)

- **Batch Job**: `strategy-governance-check`
- **역할**: 최근 N개월 백테스트 실행 후 MDD·Sharpe 열화 시 Discord 알림.  
  설정에 따라 (market, strategyType)별 **halt** 등록(TB_GOVERNANCE_HALT) → 파이프라인 실행 시 해당 조합 run 스킵.  
  Admin API: GET /api/v1/ops/governance/results, GET /api/v1/ops/governance/halts, PUT …/halts/{market}/{strategyType}/clear.

### 6.3 리스크 이벤트 알림 (장중 10분마다 09:00~15:59 평일)

- **Batch Job**: `risk-event-alert`
- **역할**: 일일 손실 한도 임박·VaR 95% 초과 검사 후 Discord 알림.

## 8. 퀀트 매매 유리 시간대 (Trading Window)

- **설정**: `investment.pipeline.trading-window.enabled`, `kr.start/end`, `us.start/end` (모두 KST).
- **기본값**: KR 09:00~10:00, US 23:30~01:00.
- **동작**: `PipelineExecutionScheduler.runNow(..., marketFilter)` 실행 시,  
  - KR run: `TradingWindowService.isInKrWindow()` true일 때만 실행.  
  - US run: `TradingWindowService.isInUsWindow()` true일 때만 실행.  
  - 윈도우 밖이면 해당 시장 run 스킵(로그: OUTSIDE_TRADING_WINDOW).

## 9. 수수료·세금 반영 (득실 순손익)

- **원칙**: 매매 판단·기대수익·손익 계산은 **수수료·세금 포함 순손익(net)** 기준.
- **FrictionCostService**: 왕복 비용률(`getRoundTripCostRate(market)`), 금액(`getRoundTripCostAmount`) 제공.  
  KR: 2×수수료+거래세+2×슬리피지, US: 2×수수료+SEC Fee+2×슬리피지(+TAF).
- **PositionSizingService**: 2:1 R:R 가정 시 기대 gross 수익률이 왕복 비용률을 초과할 때만 권장.
- **ShortTermTradingStrategyService** 등: 기대수익률·예상 수익 = gross − 왕복 비용.
- **백테스트**: `BacktestService`에서 매수/매도 시 마찰 비용 반영, `BacktestTradeDto.totalFrictionCost` 노출.
