# QA 후속 작업 가이드 (2026-03-03)

## Docker local full 기준 전체 QA 현황

| 단계 | Docker 의존 | 현재 상태 | 비고 |
|------|-------------|-----------|------|
| 1. Backend (JUnit) | 없음 (로컬 Gradle) | ✅ 통과 | 520 tests |
| 2. API 시나리오 | Backend 8080 필요 | ✅ .env 연동 | QA_* 미설정 시 investment-backend\.env 의 SUPER_ADMIN_* 자동 사용 |
| 3. Python QA (8001/8000) | data-collector 8001, prediction 8000 필요 | ✅ 통과 | Docker 풀스택 기동 시 5 checks 통과 |
| 4. Python 단위 테스트 | 없음 (로컬 py/python) | ⏭ SKIP | 9009/ModuleNotFoundError 시 SKIP 처리, pip install 후 통과 가능 |
| 5. Frontend E2E | Frontend·Backend 필요 | ⏭ 스킵 | run-full-qa 시 -SkipE2e 사용함 |
| 6. 보안 (npm audit) | 없음 | ⚠️ high 해결, moderate 잔여 | npm audit fix 적용함 |

**결론**: Docker local full만 띄운 상태에서 **스크립트가 “끝까지 에러 없이” 돌아가게** 만든 것은 맞지만, **전체 단계가 다 통과한 것은 아님.**  
- Docker 기반으로 **실제로 검증된 것**: Python QA(8001/8000).  
- Backend JUnit은 Docker 없이 로컬에서 통과.  
- API 시나리오·E2E는 스킵했고, Python 단위 테스트는 환경 이슈로 실패 가능.

**Docker local full 기준으로 “전체 QA 문제없이 통과”하려면**:  
1) API: `investment-backend\.env` 에 `SUPER_ADMIN_USERNAME`, `SUPER_ADMIN_PASSWORD` 설정 시 자동 사용(별도 QA_* 불필요).  
2) Python 단위 테스트: venv + `pip install -r requirements.txt` 후 통과; 미설치 시 SKIP 처리되어 전체 실패 아님.  
3) E2E: `-SkipE2e` 제거 후 실행해 통과 확인.  
4) 보안: high 0건 유지(이미 fix 적용됨).

## 완료된 개선

1. **run-full-qa.ps1**
   - 파싱 오류 제거: 리포트용 문자열에서 `**` 제거 → `[결과]` 등으로 대체.
   - Python 단위 테스트: try/catch 제거, `ErrorActionPreference` 조정, `python` 실패 시 `py` 런처 재시도.
   - Backend / Python QA / Python 테스트 / (선택) E2E·보안까지 한 번에 실행 가능.

2. **Backend 테스트**
   - 19건 실패 → 수정 완료. `.\scripts\run-tests.ps1` 기준 전체 통과.

3. **Python 서비스 QA (run-python-qa.ps1)**
   - data-collector 8001, prediction-service 8000 HTTP 검증. 풀스택 기동 시 통과.

4. **Frontend npm audit**
   - `npm audit fix` 실행으로 high(rollup) 해결. 남은 moderate(vitest/esbuild)는 `npm audit fix --force` 시 breaking change 가능성 있음.

## Super Admin 기준 API QA

- **run-full-qa.ps1** / **run-api-qa.ps1** 모두 **Super Admin** 계정 기준으로 동작.
- `QA_USERNAME`, `QA_PASSWORD` 를 지정하지 않으면 **investment-backend\\.env** 의 `SUPER_ADMIN_USERNAME`, `SUPER_ADMIN_PASSWORD` 를 자동으로 사용.
- Backend 기동 시 동일 .env 를 쓰므로, Super Admin 한 계정으로 로그인·API 시나리오 검증 가능.
- 비밀은 저장소에 커밋하지 말고 .env(또는 환경 변수)만 사용.

## 실행 방법

```powershell
# 전체 QA (Docker local full 기동 후, API는 .env 의 SUPER_ADMIN_* 사용)
.\scripts\run-full-qa.ps1 -SkipE2e -SkipSecurity

# API 시나리오만 수동 지정 시
$env:QA_USERNAME='...'; $env:QA_PASSWORD='...'; .\scripts\run-full-qa.ps1 -SkipE2e -SkipSecurity

# E2E 포함 (Playwright 설치 필요)
.\scripts\run-full-qa.ps1 -SkipApiScenario -SkipSecurity
```

## Python 단위 테스트 (prediction-service)

- **전제**: `investment-prediction-service`에서 `pip install -r requirements.txt` 또는 venv 활성화.
- **실행**: `py -m unittest discover -s tests -p "test_*.py" -v`
- **참고**: 로컬에 `python`/`py` 없거나 fastapi·torch 미설치 시 9009 또는 ImportError. CI/도커에서는 의존성 설치 후 동일 명령 사용.

## API 시나리오

- `plans/qa/api-qa.http` 및 `run-api-qa.ps1` 사용.
- 환경 변수 `QA_USERNAME`, `QA_PASSWORD` 설정 후 실행.

## E2E (Playwright)

- `investment-frontend`에서 `npm run e2e`.
- 브라우저 경로: `PLAYWRIGHT_BROWSERS_PATH=.playwright-browsers` (run-full-qa.ps1에서 설정).

## 보안 점검

- `investment-frontend`에서 `npm audit --audit-level=high`.
- high 이상 잔존 시 수동 검토 또는 `npm audit fix --force` (주의).
